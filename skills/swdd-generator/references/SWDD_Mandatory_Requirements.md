# SWDD 强制规范（Mandatory Requirements）

**生成或修改任何 SWDD 时必须遵守本文档，不得简化、不得遗漏、不得与源码不一致。每次执行 swdd 相关任务必须先遵守本规范。**

---

## 1. 总体原则

- **禁止简化**：不得使用 “Same pattern as…”、“omitted for brevity”、“Details omitted” 等概括性描述替代具体内容。
- **一一对应**：文档中的每个对外/对内接口、每张图都必须与源码（如 `*_int.h`、`*_prg.c`）一一对应，且逻辑一致。
- **完整覆盖**：模块内所有在 `*_int.h` 中声明的外部接口，以及文档 2.4 总览表中列出的 Unit，都必须在 2.7/2.8 中有对应的说明与**完整 Mermaid 流程图**。

---

## 2. 静态图（2.4）

### 2.4.1 Static Diagram Picture-1（Mermaid）

- **必须存在**：每个 SWDD 必须有 2.4.1 节，且包含完整的 Mermaid 图（如 `flowchart LR`）。
- **内容要求**：
  - **Callers**：列出所有调用方（如 APP、HTOS、Timer、Interrupt 等）。
  - **本模块**：列出**全部**对外接口函数（与 `*_int.h` 一致），以及文档中描述的、对理解架构有必要的内部函数。
  - **下层/BSW**：列出本模块调用的 DRTE/BSW 接口（如 DRTE_vidGetPin、LinTp、Dcm 等）。
  - **连线**：Callers → 本模块各接口；本模块 → 下层接口；若某接口内部会调用本模块其他接口（如 SetPort → enuGetPin）或下层接口，须在图中画出对应连线。
- **禁止**：静态图中不得缺少已在 2.4.2 表或 `*_int.h` 中出现的对外接口。

### 2.4.2 Component Overview Table

- **必须存在**：每个 SWDD 必须有 2.4.2 节，表格包含列：Component Name, Unit ID, Unit name, External/Internal, SW unit description, ASIL level。
- **内容要求**：表中须列出**全部** 2.7（External）与 2.8（Internal）中描述的函数/单元，Unit ID 连续且与 2.4.1 中节点一致。
- **禁止**：不得出现“仅部分列出，其余省略”的情况。

## 3. 动态图（2.6）

### 2.6 Dynamic Behavior（PlantUML）

- **必须存在**：每个 SWDD 必须有 2.6 节，且包含至少一段完整的 PlantUML 序列图（`@startuml … @enduml`）。
- **内容要求**：
  - **调用链真实**：参与者与消息顺序必须与源码调用关系一致。例如：若 APP 通过模块接口 A 再调用下层 B，则图中必须是 **APP → A → B**，不得写成 **APP → B**（越级）。
  - **周期/事件**：若模块有周期调用（如 10ms/50ms/1ms），须在图中用 `loop` 或明确消息标出周期与调用者（如 HTOS/Timer）。
  - **主要场景**：初始化、周期任务、上层读/写接口、中断/回调等主要行为路径均应在动态图中体现（可在同一张图或分图，但不得遗漏）。
- **禁止**：不得省略关键调用层级；不得将”经本模块再调下层”画成”上层直调下层”。
- **特别注意：内部函数不调用外部模块**：
  - **内部函数**（如 `BBSM_bHasVoltageFault`、`BBSM_bIsInternalTriggerActive`）本身**不调用任何外部模块**
  - 参数是由**调用者（MainFunc）获取后传给内部函数**的，不是内部函数自己获取的
  - **错误画法**：`HasFault -> SRTE : SRTE_enuGetBattLowVolErrSta()`（错误：内部函数HasFault不调用SRTE）
  - **正确画法**：`MainFunc -> SRTE : SRTE_enuGetBattLowVolErrSta()` 获取参数 → `MainFunc -> HasFault : 参数` 传入参数

## 4. 外部/内部函数（2.7 / 2.8）

### 4.0 流程图输出前必须满足（每次输出 Mermaid flowchart 前必做，避免首次生成遗漏）

**在生成或粘贴任何一段 2.7/2.8 的 Mermaid 流程图之前，必须同时满足以下三条，否则不得输出：**

1. **分支编号**：所有判断节点（菱形）的出边必须使用**数字 + 右括号 + Y/N** 标注（如 `1) Y`、`2) N`），不得使用裸的 `"Y"`/`"N"`。
   - **统一格式**：只使用 `数字+右括号+Y/N` 格式，禁止使用圆圈数字（如 ①②③）
   - **连续编号**：同一流程图内的所有分支必须连续编号（1、2、3...），每个判断的Y/N分支编号接着上一个判断的编号
   - 同一判断的两条出边使用 `-->"1) Y"|`、`"-->"2) N"|`
   - 格式示例：`-->|"1) Y"|`、`-->|"2) N"|`
   - 示例（连续编号）：
     - 第1个判断：Y分支 `1) Y`，N分支 `2) N`
     - 第2个判断：Y分支 `3) Y`，N分支 `4) N`
     - 第3个判断：Y分支 `5) Y`，N分支 `6) N`
2. **代码语言**：所有处理框（矩形）和判断框（菱形）内容必须使用**实际代码**（函数调用、赋值、条件表达式），与源码一致。
   - **禁止**使用自然语言或概括描述，如 "Find … over buffer"、"Get status"、"Handle init"、"Configure xxx"、"Set error flag" 等，无论一句还是多句。
   - **必须使用完整变量名**：`bAccelerationChangeFlagLoc`（不是`bFlag`）、`u16PeriodTime`（不是`u16Cnt`）
   - **必须使用完整函数名**：`HRTE_bGetAccelerationChangeFlag()`（不是`GetFlag`）
3. **双引号使用**：Mermaid节点文本中如果包含方括号 `[]`、圆括号 `()`、花括号 `{}`、大于小于号 `<>` 等特殊字符，**必须用双引号包裹**。
   - 示例：
     - ✅ `["bAccelFlag = HRTE_bGetAccelFlag()"]` - 圆括号需要双引号
     - ✅ `["arr[i] = func(x)"]` - 方括号和圆括号需要双引号
     - ✅ `{"arr[i] > threshold?} ` - 判断框中方括号需要双引号
     - ❌ `[arr[i] = func(x)]` - 没有双引号会导致解析错误

**自检（每输出一个 flowchart 后立即执行）：**
- [ ] 图中是否还有 `-->|"Y"|` 或 `-->|"N"|`？若有，改为 `-->|"1) Y"|` / `-->|"2) N"|` 等编号。
- [ ] 图中是否还有非代码描述（如 Find/Get/Set/Handle/Check/Update/Initialize 等泛化动词）？若有，改为与 `*_prg.c` 一致的赋值、函数调用或条件表达式。
- [ ] 节点文本中的 `[]` `()` `{}` 等特殊字符是否都用双引号包裹？
- [ ] 变量名、函数名是否与源码完全一致（没有简化）？
- [ ] **嵌套结构检查（重要）**：是否对照源码检查了if-else if-else的嵌套结构？每个else分支是否都正确连接？嵌套if之后的代码是否正确体现？
  - **特别注意独立if vs if-else**：
    - 独立if：两个或多个if语句依次执行，每个if都会被检查，可能同时触发（如IS_ARMING状态的exit1和exit2）
    - if-else：互斥结构，只能选择其中一个分支执行
    - 流程图中独立if要体现为多个独立的检查路径，if-else要用互斥结构表示
  - **特别注意每个判断分支的输出数量**：
    - if判断：必须有且只有两个输出（Y和N）
    - switch判断：可以有多个输出（case1、case2、default等）
    - 不能遗漏任何一个分支的输出
  - **特别注意编号连续性**：
    - 每个判断节点的Y和N分支必须使用连续编号（如39) Y和39) N）
    - 编号不能重复使用：一个编号只能用于一个判断节点
    - 自检验证命令：`grep -oE '\|"[0-9]+\) [YN]' file.mmd | sort`

### 4.1 每个接口必须有

- **属性表**：Prototype, Location, Function Description（必要时可加 Verification Criteria 等）。
- **Design and Implementation**：**一段完整的 Mermaid 流程图**（`flowchart TD`），描述从入口到出口的逻辑（分支、循环、调用、返回值），与源码一致。

### 4.2 禁止行为

- **禁止**用“Same pattern as xxx”“同上”“见 2.7.x”等代替本函数的流程图。
- **禁止**对内部函数（如滤波、换算）写“omitted for brevity”或“similar to xxx”；必须给出完整 Mermaid 图。
- **禁止**在表格单元格内使用未转义的 `|`，以免后续脚本（如表格解析、Word 导出）出错；可用 “OR” 等替代。

### 4.3 与源码一致

- 流程图中的判断条件、调用顺序、返回值须与 `*_prg.c` / `*_int.h` 一致；若发现文档与源码不符，以源码为准并修正文档。

## 5. 检查清单（生成/修订后必须自检）

生成或大改 SWDD 后，必须按下列项自检，任一项不满足则必须补充或修正：

- [ ] 2.4.1 静态图：是否包含本模块**全部**对外接口及必要内部节点？Callers、下层是否完整？连线是否反映真实调用关系？
- [ ] 2.4.2 表：是否与 2.4.1 及 2.7/2.8 列表一致？有无遗漏 Unit？
- [ ] 2.6 动态图：调用链是否完整（无”越级”调用）？周期与主要场景是否都体现？
- [ ] 2.6 动态图：内部函数（如HasFault、IsIntTrg）是否错误调用了外部模块？参数是否由MainFunc获取后传入？
- [ ] 2.7/2.8：每个函数是否都有**完整** Mermaid 流程图？是否存在“Same pattern”“omitted”等简化表述？
- [ ] **流程图分支编号**：每个判断分支是否均为 `-->|"1) Y"|` / `-->|"2) N"|` 等形式？是否无裸的 `"Y"`/`"N"`？编号是否连续？
- [ ] **流程图代码语言**：每个处理框/判断框是否均为实际代码（函数调用、赋值、条件）？是否无”Find/Get/Set/Handle/Configure…”等泛化描述？变量名、函数名是否完整（未简化）？
- [ ] **流程图双引号**：节点文本中包含 `[]` `()` `{}` `<>` 等特殊字符时是否都用双引号包裹？
- [ ] 表格中是否避免在单元格内使用未转义的 `|`？
